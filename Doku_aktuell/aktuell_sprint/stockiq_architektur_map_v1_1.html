9 — Engineering-Regeln (Kurzreferenz)

JavaScript (ES5-only)

Verboten: arrow =>, const/let, template literals, ?., **, async/await
Verboten: nicht-ASCII in Script-Bloecken
Null-Guard: var _sn = s.n ? s.n.toLowerCase() : ''
HTML-Edits: str_replace mit exaktem Anker, nie Positional-Slicing
Versionsstring: str_replace via Ccode (NICHT PS-Regex)

Python

tk.history() statt yf.download()
sanitize() auf alle yfinance-Werte
encoding=utf-8 bei allen Dateioperationen
Kein nicht-ASCII in print()
PS-Mehrzeiler: @'...'@ | Out-File script.py → python script.py

Schritt-0-Pflicht (vor jedem Brief)

Output nach debug_s0.txt (min 800 Zeichen, Ccode direkt)
Zwei-Stufen: Diagnose lesen → dann Brief schreiben
JSON-Root pruefen: scores['scores'] = LIST

11 — Architektur-Prinzipien (AP-1 bis AP-5)

Abgeleitet aus Post-Mortem Sprint 31/32: STOCKS-Array stale auf GitHub Pages (37% Signale falsch). Behoben Sprint 32 — 264/264 FAIL:0 verifiziert. Prinzipien gelten dauerhaft.

AP-1 — Single Source of Truth

Fuer jedes Datenfeld genau eine authoritative Quelle.
signal, sw, hist, price, score: scores.json via _scoresIdx
STOCKS[] = Anzeige-Cache, NICHT Datenquelle.
Kein Feld darf aus zwei Quellen parallel befuellt werden.

AP-2 — Sync explizit und vollstaendig

STOCKS als Cache: alle signal-relevanten Felder synchronisieren.
Kein implizites "wird schon stimmen", kein partieller Update.
Neues Feld in score.py → Sync-Liste sofort aktualisieren.
Aktuell synced: sw / above200 / p / hist / rsi / div

AP-3 — Stille Fehler sind kritischer als laute

loadFund() scheitert auf GitHub Pages lautlos (FD gitignored).
Ergebnis: maximale Divergenz, kein diagnostisches Signal.
Massnahme: console.warn wenn FD leer (low priority, noch offen).
Leitprinzip: lieber Crash als stiller Datenfehler.

AP-4 — Tests muessen Produktions-Environment simulieren

Mindestens ein Test-Szenario: GitHub-Pages-Zustand (kein FD, nur scores.json).
loadFund() mit leerem FD testen → Block10 via page.evaluate().
Lokal besteht != Produktion korrekt.

AP-5 — Konsistenz-Assertion nach Datenlade-Zyklus

Nach loadScores()-Callback: automatische Pruefung _scoresIdx vs. STOCKS fuer kritische Felder (sw, hist, signal).
Implementiert als Block9 Puppeteer-Test — Akzeptanzkriterium: ≥95% Konsistenz.
Status: AKTIV — 40 Puppeteer-Tests inkl. Block9/Block10.
